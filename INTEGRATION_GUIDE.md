# Integrating Your Own Datasets — Address Standardization System

This guide covers everything needed to replace the project's synthetic demo dataset
with your two real datasets (`address-table1.csv`, `address-table2.csv`), retrain the
LightGBM confidence model, and deploy it — without breaking the REST API, review
queue, or audit trail.

## 0. What you're starting with

| File | Columns | Quality notes (measured on your actual data) |
|---|---|---|
| `address-table1.csv` | `county_name, street_nbr, street_name, zip5` | 155,453 rows. **44,876 exact duplicates** (29%). 5,710 rows with unusable house numbers (ranges like `75-81`, junk like `OFF`, Excel-corrupted values like `09-Nov`). 24,499 rows (~16% of cleaned) missing `zip5`. **No city or state column at all.** |
| `address-table2.csv` | `city, countyOrParish, stateOrProvince, address, shortAddress` | 16,105 rows. 258 exact duplicates. 3 sentinel rows (`zip 99999`, city `Out-of State`) — not real addresses. `address` is otherwise clean and human-verified — it's used as ground truth. Contains lot-style rural addressing (`L6.21 Stony Creek Rd`) and embedded units (`# F144`, `Apt 2`, `Lot 7`). |

Because table1 has no city/state, the pipeline below **imputes** city for table1 rows
from table2's zip→city / county→city mapping, and assumes state = NY. This is a
necessary approximation, not verified data — it's logged so you know which rows it
affected.

## 2. Ingest both CSVs into PostgreSQL

```bash
# Start the stack if it isn't running
docker compose up -d postgres

# Apply the new migration (skip if this is a brand-new DB — docker-entrypoint-initdb.d
# already ran every file in database/migrations/ automatically on first boot)
docker compose exec -T postgres psql -U addruser -d address_db \
  < database/migrations/003_source_datasets.sql

# Run the ingestion script (from the project root, on your host machine or
# inside any container that has network access to postgres)
pip install psycopg2-binary pandas --break-system-packages   # if not already installed
export DATABASE_URL=postgresql://addruser:addrpass@localhost:5432/address_db
python scripts/ingest_datasets.py
```

What this does:
- Loads **every** row from both CSVs verbatim into `staging_table1_county_street` /
  `staging_table2_city_address` (your audit copy of "what we received", duplicates and
  junk included).
- Cleans each (drops exact duplicates, rows with unusable house numbers, sentinel rows)
  and inserts the survivors into the production `raw_addresses` table with
  `source_system = 'table1_county_street'` or `'table2_city_address'`, so every row
  remains traceable back to its source file.
- Writes a summary row to `ingestion_log` and prints a JSON summary to stdout.

This step makes your data visible to the backend's existing `GET /addresses` /
review-queue endpoints — it does **not** call the ML model yet. To actually run
inference + routing on these ingested rows, call `POST /addresses/standardize` for
each `raw_addresses.raw_text` (a simple loop over the table, or extend
`ingest_datasets.py` with an HTTP call if you want this automated — intentionally left
manual since it's a one-line addition once you decide whether to do it inline or via a
queue).

## 3. Build the labeled training pairs

```bash
cd ml-service
python data/prepare_training_data.py
```

This reads `data/raw/address-table1.csv` and `data/raw/address-table2.csv` and produces:

| Output | Rows (on your data) | Purpose |
|---|---|---|
| `data/training_data.csv` | 117,966 | Fed into `train.py` / the `/retrain` endpoint. **85%** of all generated pairs. |
| `data/validation_holdout.csv` | 20,817 | **Never** touched during training. Used by `validate.py` to measure true generalization. **15%** of all generated pairs, stratified-shuffled with `random_state=42`. |
| `data/data_quality_report.json` | — | Exact counts of every row dropped, deduped, or imputed, and why. |

**How the pairs are built (so you can audit/tune the logic):**
- **table2** rows: the `address` column is treated as ground truth (canonical). A noisy
  "raw" counterpart is synthetically generated from it (abbreviated suffixes/directions,
  dropped ZIP ~20% of the time, case changes, occasional comma removal, ~12% typo rate)
  → label `1`.
- **table1** rows: house number is taken from `street_nbr` (first integer in the field,
  so `"75-81"` → `75`). Street name is expanded from its abbreviated form (`"OAKWOOD DR"`
  → `"Oakwood Drive"`). City is imputed (zip match against table2 first, then county
  match, then a random NY city as last resort) and state is set to `NY`. The **raw**
  side of the pair is exactly what the source gave you — abbreviated street, no city, no
  zip — because that's the real shape of input this system needs to standardize.
- **Negative pairs** (label `0`, ~18,000 of them): a raw address from one record is
  paired with the canonical address of an unrelated record, so the model also learns to
  reject wrong matches, not just normalize correct ones.

Re-run this script any time you update the raw CSVs.

> **Tuning knobs** (top of `prepare_training_data.py`):
> `HOLDOUT_FRACTION` (default 0.15), `NEGATIVE_PAIR_FRACTION` (default 0.15), and the
> noise probabilities inside `noisify_address()` (abbreviation rate, ZIP-drop rate, typo
> rate). Raise the typo rate if your production inputs are noisier than this; lower the
> ZIP-drop rate if your real pipeline usually has a ZIP.

## 4. Extract features and train LightGBM

Feature extraction (`app/features.py`) and parsing (`app/parser.py`, via **libpostal**)
both require the compiled libpostal binary, which only exists inside the `ml-service`
Docker image (see its `Dockerfile`) — **run training inside that container**, not on
your host, unless you've separately installed libpostal locally.

```bash
docker compose build ml-service        # rebuild if you haven't already
docker compose up -d ml-service
docker compose exec ml-service python train.py --register
```

`train.py`:
1. Loads `data/training_data.csv` (117,966 rows from step 3).
2. For every row, parses the raw address with libpostal, rule-standardizes it
   (`app/standardizer.py`), and extracts the 8 features in `FEATURE_ORDER` (Jaccard
   similarity, edit-distance ratio, length-diff ratio, has-zip, has-house-number,
   zip-in-raw, directional-expanded, suffix-expanded).
3. Splits that feature set **80/20** (train/internal-validation, `random_state=42`,
   stratified by label) — this is the split `retrain_model()` already did before your
   data was added; nothing changed there.
4. Trains an `LGBMClassifier(n_estimators=100, learning_rate=0.05, max_depth=5)`.
5. Saves the pickle to `models/lgbm_model.pkl` (mounted volume — persists outside the
   container).
6. With `--register`, also writes a row into the Postgres `model_registry` table and
   marks it `is_active = TRUE` (deactivating whichever model was active before), exactly
   like the backend's own `/retrain` flow does.

**Train vs. validation, explicitly:**
- **Training**: 80% of `training_data.csv` → 94,372 rows.
- **Internal validation** (reported by `train.py`'s printed metrics): 20% of
  `training_data.csv` → 23,594 rows. This is *held out from training* but was generated
  by the *same* `prepare_training_data.py` run, so it shares the same noise
  distribution as the training rows.
- **External validation** (`validate.py`, next section): the separate 20,817-row
  `validation_holdout.csv` that `prepare_training_data.py` set aside *before* any
  training happened — the model has never seen any row derived from these source
  records in any form.

### Parameter tuning
Edit `app/retraining.py`:
```python
clf = LGBMClassifier(
    n_estimators=100,     # more trees = better fit, slower train/predict, risk of overfit
    learning_rate=0.05,   # lower = more conservative, usually pair with more estimators
    max_depth=5,          # shallower = less overfitting on ~118k rows of mostly-similar features
    random_state=42,
)
```
With 8 simple numeric features and >100k rows, this model is unlikely to need deep
trees — if you see overfitting (training accuracy near 1.0, holdout accuracy much
lower), reduce `max_depth` or `n_estimators` first.

## 5. Validate on the untouched holdout set

```bash
docker compose exec ml-service python validate.py
```

Prints overall accuracy/precision/recall/F1/confusion-matrix plus a breakdown **by
source** (`table1`, `table2`, `synthetic_negative`), so you can see directly whether
the model generalizes worse on table1 (lower-confidence labels, due to imputed
city/state) vs. table2 (verified ground truth). If table1 metrics are notably worse,
that's expected — treat it as a signal to collect real city/state ground truth for
those records rather than a model bug.

## 6. Deploy the retrained model

The model artifact lives at `ml-service/models/lgbm_model.pkl`, which is a
docker-compose **volume mount** (`./ml-service/models:/app/models`) — so once
`train.py` finishes, the new model is already live:

```bash
# model.py lazily loads from MODEL_PATH on first prediction after process start;
# restart ml-service to force-reload immediately rather than waiting for the
# in-memory cache to be replaced by the next /retrain call:
docker compose restart ml-service

curl http://localhost:8000/model/metrics       # confirm new metrics are served
curl http://localhost:3000/health              # confirm backend still sees ml-service healthy
```

Nothing in the NestJS backend (`addresses.service.ts`, `review.service.ts`,
`audit.service.ts`) needed to change:
- Confidence routing (`AUTO_ACCEPT_THRESHOLD=0.80`, `REVIEW_THRESHOLD=0.50` in `.env`)
  reads whatever `confidence_score` the ML service returns — it doesn't care which
  model produced it.
- The review queue, feedback table, and audit trail all key off
  `standardization_results` / `raw_addresses`, which are populated the same way
  regardless of dataset.
- Going forward, human corrections collected through the review queue flow into
  `POST /retrain` exactly as before (`MlService.triggerRetrain()` →
  `MlClientService` → ml-service `/retrain` → `retrain_model()`), which now combines
  your new `training_data.csv` base with live feedback.

If you want stricter/looser auto-accept behavior given the new model's confidence
distribution, tune `AUTO_ACCEPT_THRESHOLD` / `REVIEW_THRESHOLD` in `.env` and restart
`backend`.

## 7. Monitoring model performance over time

- `GET /model/metrics` (ml-service) — last training run's accuracy/precision/recall/F1.
- `GET /addresses` (backend) + `review_queue` table — watch the proportion of
  `auto_accepted` vs. `pending_review` vs. `flagged` over time; a rising `flagged` rate
  on real production traffic usually means the input distribution has drifted from what
  the model was trained on.
- `model_registry` table — every training run via `train.py --register` or the
  `/retrain` endpoint is logged here with its metrics, so you can compare versions.
- `audit_trail` table — every individual standardization + every `model_retrained`
  event, for full traceability of "which model produced this prediction."

## 8. Troubleshooting common data quality issues

| Symptom | Cause | Where it's handled |
|---|---|---|
| House number like `75-81`, `1100-1104` | Source data uses ranges for multi-lot parcels | `clean_street_nbr()` takes the first number in the range. If you'd rather expand ranges into multiple rows, modify that function. |
| House number like `09-Nov` | Excel auto-converted a typed value like `"9-Nov"` (originally probably `9-11`) into a date | Currently dropped (no leading digit to extract) — counted in `data_quality_report.json` under `rows_with_unusable_house_number_dropped`. Re-export from the source system as CSV without going through Excel to avoid this. |
| `zip5 = "99999"` or city = `"Out-of State"` | Sentinel/placeholder values for out-of-state or unresolved addresses | Filtered out in both `prepare_training_data.py` and `ingest_datasets.py` (`sentinel_or_out_of_state` count). |
| Large duplicate counts (29% in table1) | Repeated submissions / multiple records per property over time | Dropped via `drop_duplicates()` before any pairing/insertion — exact duplicates only; near-duplicates (e.g. typo variants of the same address) are *not* deduped, since the model needs to learn from those. |
| Missing `street_name` | Incomplete source records | Dropped — there is no way to standardize an address with no street. |
| Missing `zip5` (~16% of cleaned table1) | Geocoding/zip lookup never completed at the source | Kept — these become realistic "raw" training inputs that lack a zip, which is exactly the kind of input the model needs to handle gracefully. The `has_zip` / `zip_in_raw` features exist specifically for this. |
| Table1 has no city/state | Source system simply doesn't track it at this table | **Imputed**, not authoritative — see step 3. If you can join this dataset against a parcel/tax-roll system that does have city/state, do that instead and re-run `prepare_training_data.py` with real values; the imputation is a fallback, not a recommendation. |
| `psycopg2.errors.UndefinedColumn` or similar during `ingest_datasets.py` | Migration `003_source_datasets.sql` wasn't applied | Run the `psql` command in step 2 against the right database. |
| `ON CONFLICT DO NOTHING` silently does nothing useful on re-runs | `raw_addresses` has no unique constraint on `(source_system, source_record_id)` by default | Add one if you intend to re-run ingestion repeatedly: `ALTER TABLE raw_addresses ADD CONSTRAINT uq_raw_addr_source UNIQUE (source_system, source_record_id);` |
| `train.py` / `validate.py` fail with `ImportError: No module named 'postal'` | Running outside the `ml-service` container, where libpostal isn't compiled/installed | Always run these two scripts via `docker compose exec ml-service python ...`, or install libpostal + `pip install postal` locally first. |
